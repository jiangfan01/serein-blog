## 2. 后端判断力：邀请码 bug 实战

**bug**：AI 写的 `InviteCode` 自相矛盾——关系是 1:1（`User.inviteCodeId @unique`），却带 `maxUses`/`usedCount`（暗示一码多人 1:N）。`maxUses=5` 时第二个人注册会撞唯一约束，失败。

**怎么判断 1:1 vs 1:N（一辈子能用的技能）**：
- 外键(FK)在哪张表？看 `@relation(fields:[...])` 写在谁身上。
- FK 带 `@unique` → 1:1；不带 → 1:N。
- 反向字段（如 `usedBy User?`）不是数据库列，只是 Prisma 给的虚拟关联字段。

**修复**：业务拍板"一码一人"→ 删 `maxUses`/`usedCount`，用 `usedBy` 关系当"是否已用"的唯一真相来源。注册逻辑从"事务(建用户 + usedCount+1)"塌缩成"单次写入（建用户时设 inviteCodeId）"，并加 `P2002`（唯一约束冲突）catch 做并发兜底。

**审后端代码的四条原则**：
1. 先问关系（1:1/1:N/N:N），再看字段。骨架错了字段再美也是危房。
2. 用数据流推演，不靠感觉（"建码→A注册→B注册"脑里跑一遍它在哪步炸）。
3. 字段语义要和表结构对齐（出现 `maxUses` 这种词，立刻查关系是不是真 1:N）。
4. 永远多想一步并发（凡是"读出来→判断→写回去"的计数器/库存/余额，问一句两个人同时来会怎样）。

**两个关键区分**：
- 防一码二用 → 靠数据库 `@unique`（约束就是并发锁）。
- 保证多步操作原子性 → 靠事务 `$transaction`。这俩是两件事。
- **一个事实只存一个地方**：能从关系推出来的状态，别再单独存 `status` 字段（否则两份会打架）。

> ⏳ 遗留：Supabase 真库 `db push` 没跑（直连 5432 不通 + 等迁服务器）。代码与库当前一致、能跑（库里多两列死字段无害）。删列等迁京东云时顺手做。

## 3. L1 · 纯图机制（不碰 LLM）

文件：`01-minimal-graph.ts` / `.html`

**一句话**：LangGraph = 一张**可以有环的有向图**；图运行时从 START 驱动，数据装在 State 里流动，节点执行逻辑返回局部更新（reducer 合并），边（尤其条件边）读 State 决定下一步，直到 END。

**四件套**：
- **State**：全图共享的"内存"。被动数据，不含逻辑。
- **Node（节点）**：干活的函数。输入完整 State，返回**对 State 的局部更新**（不是整个 State）。
- **Edge（边）**：固定走向。
- **Conditional Edge（条件边）**：读 State 决定走哪条路。

**reducer（最核心、最易忽略）**：节点 return 字段不是"覆盖"，而是按该 channel 的 reducer "合并"。
```ts
const StateAnnotation = Annotation.Root({
  value: Annotation<number>(),                      // 无 reducer = 默认"覆盖"
  log: Annotation<string[]>({
    reducer: (prev, next) => [...prev, ...next],     // "追加"
    default: () => [],
  }),
});
```
→ 跑完 `value` 只剩最后一个值（覆盖），`log` 留下全部（追加）。
→ 对照 agent.ts：`messages` append、`route` replace、`inputTokens` sum，三种 reducer。
→ **坑**：把 value 改成累加 reducer 时，节点内部已经算过的值会被再加一遍 → reducer 语义必须和节点返回值配套设计。

**compile()**：把"图纸"（addNode/addEdge 的静态描述）冻结成"能跑的引擎"。做三件事：① 校验结构（没入口/有孤儿节点就报错）② 装上执行引擎 ③ 返回带 `.invoke()/.stream()` 的对象。≈ Eino 的 `NewRunner`。删掉 `START→x` 的边，报错发生在 compile（校验期），不是 invoke。

**解剖图（文字版，对应 .html 里的编号图）**——一张真实 agent 图：
```
        START①
          │ 普通边(实线，固定)
        agent② ──tool_calls?──┐
          │  条件边(虚线，读State决定)
        否↓                   有↓
        END⑥                tools④
          ↑__________环⑤(回边)__│   ← tools 干完回到 agent
State⑦ 贯穿全程，所有节点都读它改它
```
- ①⑥ 边界节点 START/END：入口出口，内置。
- ②④ 节点：干活函数。
- ③ 条件边：读 State 分叉（`shouldContinue`）。
- ⑤ 环：回边，Agent 反复"调工具→再想"的灵魂，也是它**不是 DAG** 的原因。

## 4. L2 · 会调 LLM 的节点

文件：`02-llm-node.ts` / `.html`（真调 DeepSeek）

**骨架和 L1 一样**，只是节点内部换成调模型：
```ts
async function agentNode(state) {
  const response = await model.invoke(state.messages);  // 返回一条 AIMessage
  return { messages: [response] };                       // append reducer 接到历史后
}
```
- `messages` channel 用 append reducer 累积对话（装的是消息对象，不是字符串）。LangGraph 内置 `MessagesAnnotation` 省得自己写。
- 流式：`model.stream()` 吐 token，前端逐字渲染。
- **两条不同的线别混**：写进 State（喂给下游节点，数据流）vs 流式推给前端（喂给屏幕，展示流）。两者并行但目的不同。
- 🔒 安全铁律：API key 绝不进浏览器前端（F12 就被偷）。真实调用在服务端，key 留 `.env`。

## 5. L3 · ReAct 循环（手搓 createReactAgent）

文件：`03-react-loop.ts` / `.html`。这就是 agent.ts 的核心。

```ts
const graph = new StateGraph(StateAnnotation)
  .addNode("agent", agentNode)          // L2 的 LLM 节点（model.bindTools(tools)）
  .addNode("tools", toolNode)           // 内置 ToolNode，自动执行 tool_calls
  .addEdge(START, "agent")
  .addConditionalEdges("agent", shouldContinue, { tools: "tools", [END]: END })
  .addEdge("tools", "agent")            // ← 回边 = 环
  .compile();

function shouldContinue(state) {        // = agent.ts 的 shouldContinue
  const last = state.messages.at(-1);
  return last.tool_calls?.length ? "tools" : END;
}
```
**实测**：问"杭州天气+穿什么"→ 第1次LLM(决定调get_weather) → 工具执行 → 结果回喂 → 第2次LLM(看着天气给建议)。4 条消息，2 次 LLM 调用。

**灵魂**：工具不产生决策，它只把信息塞回 State；真正的"下一步"永远是 LLM 看着新结果重新推理出来的 → 所以工具调用必然伴随又一次 LLM 调用。这就是环存在的意义。

**`bindTools(tools)`**：把工具的 description（写给 LLM 看的说明书）告诉 LLM，它才有能力在回复里带 `tool_calls`。description 调优 = 让 LLM 准确判断"何时该调我"。

**防无限循环（三档，由低到高）**：
1. 硬兜底：`recursion_limit`（LangGraph 默认 25 super-step）超了抛 `GraphRecursionError`。防失控，但崩给用户看，体验差。
2. 优雅收尾：接近上限时注入指令"不能再调工具了，用现有信息给最终答案"，逼 LLM 收敛。
3. 总结续接（上家 Eino 做法，最高级）：接近上限→让 LLM 总结进度→干净结束→自动开新一轮带摘要接着干（`ContinueAgentStream`）。
- 补充：无限循环的真实病因常是某工具一直报错、LLM 反复重试 → 也要检测"重复的相同 tool_call"。

## 6. L4 · 意图分类 + 动态模型路由

文件：`04-routing.html`、`04b-intent-classify.html`。对应 `router/` 目录。

**为什么路由**：钱和延迟。"你好呀"用 reasoner 模型 = 烧钱还慢。让每个请求用刚好够用的模型。路由规则放数据库（`RoutingRule` 表）→ 改路由不改代码、不重启（热更新）。

**意图识别 = 三级瀑布（越往下越贵，能上级解决绝不下沉）**——`intent-classifier.ts`：
```
classify(question):
  ① 缓存：cache.get(key) 命中 → 直接返回      [0ms / 免费]
  ② 规则：quickMatch 关键词匹配               [~0ms / 免费]
       置信度 >= 0.9 → 拍板，不调 API
  ③ 小模型：deepseek-flash 输出 JSON          [~200ms / 花钱]
       {intent, confidence, reasoning}
  ④ 失败 → 降级回 ② 的规则结果
```
- 五种意图：`simple_chat / rag_query / web_search / complex_reasoning / code_analysis`。
- "AI 意图识别厉害"= 第③级小模型在做分类，配①②兜住大部分简单情况省钱。
- review 眼：那个 cache 是进程内 `Map`（重启丢、多实例不共享、无上限）→ 生产换 Redis + TTL + 容量上限。

**架构顿悟**：我凭直觉想"设计"的错误日志表（记录调了什么模型/工具、什么错、token），库里早有了 = `ModelCallLog`（status/errorMessage/usedFallback/fallbackReason/toolCalls/tokens/latency）+ `ModelHealthCheck`。映射轴是**意图→模型**（`RoutingRule`），不是工具→模型。
→ 拐点信号：能凭直觉重新发明出自己系统已有的设计 = 从"AI 写的"变"我懂的"。

## 6.5 L5 · checkpointer 断点续传 / 跨调用记忆

文件：`05-checkpointer.ts` / `.html`。回答"State 存哪"。

**一个开关**：`compile({ checkpointer })`，State 从"即焚"变"按 thread_id 长存"。
```ts
const checkpointer = new MemorySaver();           // 开发用；生产用 PostgresSaver
const graph = new StateGraph(...).compile({ checkpointer });
await graph.invoke({ messages: [...] }, { configurable: { thread_id: "user-serein" } });
```
**每次 invoke 的内部动作**：① 按 thread_id 取出上次 state → ② append 新消息 → ③ 跑图 → ④ 新 state 存回该 thread_id。
- 同 thread_id = 记忆累积；换 thread_id = 另一份独立存档（会话隔离的键）。
- `graph.getState(config)` 可读某 thread 的存档（断点续传/调试）。
- MemorySaver 存内存（重启丢）；PostgresSaver/RedisSaver 跨进程跨重启。

**两种记忆实现的取舍（面试爱问）**：
- 手动派（serein-blog 现状）：消息落 `ChatMessage` 表，每次 `buildContext` 重新喂。灵活、可精细控制塞什么。
- 框架派（checkpointer）：一个参数白送跨调用记忆 + 断点续传 + 时间旅行 + human-in-the-loop。省事。
- 你 `ChatExecution`(status/progress) = 手搓 checkpoint；Eino `CheckPointStore`+`WithCheckPointID` = 框架级。三种都摸过。

## 7. 横向核心概念（必背）

- **Agent 本质**：给"不确定的 LLM"套上确定的契约——控制输入（上下文工程）、约束输出（结构化/工具 schema）、编排流程（图+状态机+循环边界）。（= 你截图那句"用 AI 控制写代码，我控制状态、边界、流程"）
- **State 生命周期**：默认活在内存、仅一次执行期间，跑完即焚。要长存两条路：① 手动落库（serein-blog：`ChatMessage` 表 + `buildContext` 每次重新喂）② 框架 checkpointer（Eino：`CheckPointStore` + `thread_id`）。
- **State ≈ zustand？** 对一半。相似：共享 store、读它改它、reducer。区别：① 生命周期——zustand 浏览器长存，LangGraph State 后端一次执行即焚；② 更新方式——zustand 命令式 `set`，LangGraph 声明式（节点 return 局部更新，reducer 合并，更像 Redux）。
- **ReAct**：Reasoning + Acting，工具结果回喂触发再推理。

## 8. 面试黑话合集（背下来能镇场）

- "LangGraph 和传统 DAG 工作流引擎（Airflow）最本质区别是**支持循环**。Agentic loop 必须靠环表达'反复调工具直到完成'。"
- "状态更新靠每个 channel 的 reducer：messages 用 append、route 用 replace、token 用 sum，比手动拼数组不易错。"
- "我把 LLM 调用封装成图节点，输入输出走 messages channel，让它和工具执行、路由判断处在同一个可编排、可观测、可恢复的执行模型里。"
- "ReAct agent 是带条件出口的循环图，终止条件是 LLM 不再请求工具，所以必须加 recursion_limit 兜底防死循环。"
- "我对循环做三层防护：recursion_limit 硬兜底 + 接近上限强制收敛 + 长任务总结续接；还检测重复 tool_call（无限循环常是工具反复报错）。"
- "意图分类三级降级：缓存→关键词规则→小模型，规则层置信度阈值 0.9，高于阈值直接返回避免 API 调用。在准确率和成本间分层取舍。"
- "邀请码单次消费我用数据库唯一约束做并发兜底，而不是应用层加锁——把一致性下沉到存储层。"
- "模型路由把'用哪个模型'从硬编码变成数据库可配置 + 热更新 + 可降级，便宜问题别浪费贵模型。"

```
learn/                         ← 已挪到项目根目录
├── 00-学习进度.md             ← 本文件（行动清单在最上面）
├── 01-minimal-graph.ts        纯图机制
├── 01-minimal-graph.html      图机制可视化 + 零件解剖图
├── 02-llm-node.ts             会调 LLM 的节点（真调 DeepSeek）
├── 02-llm-node.html           token 流入 State 可视化
├── 03-react-loop.ts           ReAct 循环（真跑，看 2 次 LLM 调用）
├── 03-react-loop.html         单步看环转一圈
├── 04-routing.html            意图→路由→选模型→降级 可视化
├── 04b-intent-classify.html   意图识别三级瀑布 可视化
├── 05-checkpointer.ts         断点续传（真跑，看换 thread_id 记不记得）
├── 05-checkpointer.html       两个 thread 各存各的 可视化
└── algo/                      算法轨
    ├── 01-two-pointers.ts/.html        双指针·对撞
    └── 02-fast-slow-pointers.ts/.html  快慢指针·链表
```
- 跑 .ts：`pnpm exec tsx learn/01-minimal-graph.ts`（在项目根目录跑，读 .env 的 DEEPSEEK_API_KEY）
- 看 .html：`open learn/01-minimal-graph.html`（或浏览器直接拖入）

> 复习顺序：先读本 md 对应章 → 跑/点对应文件加深手感 → 用"面试黑话"自测能不能讲出来。
