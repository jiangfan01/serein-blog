# 图书自动化资源生产：从整本 PDF 到可审核内容树

这篇记录我最近在做的“图书自动化资源生产”方向。

背景很现实：出版社给到的资源经常是一整本 PDF，里面可能有目录、章节、单元、课文、诗词、练习题。人工要从 PDF 里一段段复制、命名、上传、生成音频、再挂到图书资源下，效率非常低。

所以我们想做的不是“一键完美生成整本书资源”，那不现实。更合理的目标是：

> 系统先自动完成 50% - 80% 的重复工作，再交给运营或出版社人工审核修正。

这篇主要讲我对这套系统的拆解方式。

## 为什么这个需求难

如果只是上传 PDF，然后把文字提取出来，这件事不算最难。

真正难的是：

1. 怎么知道目录在哪里；
2. 怎么理解目录层级；
3. 怎么把目录页码和真实 PDF 页面对应起来；
4. 怎么判断哪些节点是章节，哪些只是大纲；
5. 怎么把每个章节对应的正文准确提取出来；
6. 怎么让人工可以编辑、确认、追责；
7. 怎么把最终内容变成 PDF、音频、视频等资源。

一本书在业务上看起来像这样：

```
书
  → 年级 / 分册 / 模块
    → 单元
      → 课文 / 章节 / 诗词 / 文章
        → 正文内容
```

但 PDF 解析出来可能是这样：

```json
[
  { "text": "第一单元·植物的生长变化", "text_level": 1 },
  { "text": "1种子里孕育着新生命 2种植凤仙花", "text_level": 1 },
  { "text": "奇异的植物", "text_level": 1 },
  { "text": "世界上的植物千姿百态...", "text_level": null }
]
```

解析工具只能告诉你“它看起来像标题”，但它不知道业务上哪个节点应该生成资源。

所以这个需求的本质不是 PDF 解析，而是：

> 从半结构化 PDF 里，还原出一本书真实可用的内容树。

## 核心思路

我现在更认可的链路是：

```
上传整本 PDF
  ↓
MinerU 解析原始文本 / 页码 / 块信息
  ↓
提取目录候选内容
  ↓
LLM 根据目录还原 L1 / L2 / L3 内容树
  ↓
系统根据页码和页面文本挂载正文
  ↓
入库为 draft
  ↓
人工审核、编辑、拖拽、删除、补充
  ↓
确认 confirmed
  ↓
异步生成 TTS / PDF / 视频等资源
  ↓
挂载到图书资源表
```

这里要注意一个关键点：**LLM 不负责从零处理整本 PDF**。

更合理的方式是让传统解析工具先把原始材料拿出来，然后让 LLM 做它擅长的事：理解目录、还原层级、纠正结构。

## 第一阶段 MVP

第一阶段不追求一步到位，只验证最关键的链路：

- 上传整本 PDF；
- MinerU 解析出原始文本；
- 提取目录相关内容；
- LLM 生成目录树；
- 前端渲染树；
- 人工可以编辑树；
- 后端保存草稿；
- 确认后进入资源生成。

也就是先把“书的结构”搞定。

因为只要结构树对了，后面所有事情都会简单很多：

```
内容树节点
  → 作为资源名称
  → 绑定正文文本
  → 生成音频
  → 生成章节 PDF
  → 生成视频素材
  → 挂载到图书章节
```

真正的核心资产不是音频文件，也不是 PDF 文件，而是这棵被人工确认过的内容树。

## 数据结构设计

我会把中间态设计成一张内容树表，例如：

```typescript
interface BookContentTreeNode {
  id: number;
  bookId: number;
  parentId: number | null;
  title: string;
  level: 1 | 2 | 3;
  sortOrder: number;
  pageStart?: number;
  pageEnd?: number;
  text?: string;
  status: 'draft' | 'confirmed' | 'producing' | 'done' | 'failed';
  audioUrl?: string;
}
```

### 为什么不能只存在前端内存

因为这不是一个临时 UI 状态，而是一份需要反复审核、修改和追踪的数据。

正确流程应该是：

```
解析完成
  → 后端直接入库，status = draft

前端进入审核页
  → GET /tree/{bookId}
  → 渲染数据库里的树

运营编辑
  → 保存草稿
  → 更新数据库

运营确认
  → status = confirmed

生成资源
  → status = producing / done / failed
```

前端可以维护临时编辑状态，但最终可信数据必须在数据库。

### 前端 ID 和数据库 ID 要分开

树编辑器里新增节点时，前端通常会生成临时 ID：

```typescript
const id = `draft-${crypto.randomUUID()}`;
```

但数据库里的 `parent_id` 是自增数字 ID。这里必须做一层映射：

```
前端节点 ID
  → 入库后生成数据库 ID
  → 建立 frontendId -> dbId 映射
  → 第二遍回填 parent_id
```

否则就会出现一个很隐蔽的问题：节点都入库了，但父子关系丢了，整棵树变成平铺列表。

## 为什么要引入 LLM

一开始我也想过用正则和算法硬解析目录，比如：

- 匹配“第一单元”；
- 匹配“第 X 章”；
- 匹配页码；
- 根据缩进和标题长度推断层级。

但书的格式太多了：

```
七年级（上）
  古代诗歌四首
    观沧海 / 曹操
    次北固山下 / 王湾

第一单元·植物的生长变化
  1 种子里孕育着新生命
  2 种植凤仙花

模块一
  项目任务
    活动 1
```

只靠正则很容易越写越重，而且规则之间互相影响。

LLM 更适合做这一层：

> 给它目录原文和少量结构要求，让它输出标准 JSON 树。

比如 Prompt 目标可以非常明确：

```text
你需要从教材目录中还原内容树。

规则：
1. 只输出 JSON。
2. 最多保留 L1 / L2 / L3 三层。
3. 目录本身不是节点。
4. 页码必须来自原文，不允许编造。
5. 如果某一项没有页码，pageStart 设为 null。
6. 保留原始标题，不要过度改写。
```

这样 LLM 做语义理解，系统做持久化、校验、生成资源，边界会清楚很多。

## 人工审核为什么必须存在

出版社内容不能出错，尤其是教材、教辅、诗词、课文这些资源。AI 和算法再强，也不应该直接把结果发布出去。

所以我认为这套系统必须把人工审核放在核心流程里：

```
AI 生成内容树
  → 人工检查目录层级
  → 人工检查标题
  → 人工检查正文
  → 人工确认
  → 系统生成资源
```

这不是产品能力弱，而是业务上必须可控。

系统真正节省的是这些重复劳动：

- 不用人工一页页找目录；
- 不用人工逐个输入资源名称；
- 不用人工复制大段正文；
- 不用人工一个个调 TTS 接口；
- 不用人工手动上传 OSS 再关联资源。

人工最终只需要做审核和少量修正。

## 后续怎么扩展成工具箱

当内容树这层跑通后，后续能力其实可以工具化：

```
内容树
  ├─ 生成章节 PDF
  ├─ 生成 TTS 音频
  ├─ 生成讲解视频脚本
  ├─ 生成题目解析
  ├─ 生成知识点摘要
  └─ 绑定到图书资源
```

这些工具可以独立计费，也可以作为 B 端 SaaS 的增值能力。

我理解的产品闭环是：

```
出版社上传整本书
  → 系统解析并生成内容树
  → 人工审核确认
  → 调用不同工具生成资源
  → 资源回写到图书管理
  → 小程序端消费
  → B 端按工具或积分付费
```

这比单独做一个“AI 生成内容”更现实，因为它不是让 AI 接管全部流程，而是把 AI 嵌进确定性的业务链路里。

## 阶段性结论

这套系统最难的点不是 TTS，也不是上传 OSS，而是中间这棵内容树。

我现在对它的理解是：

> 内容树是整本书自动化生产的中间表示。

只要这棵树足够接近真实书本结构，后面的资源生产就会变成相对确定的数据操作。

所以第一阶段应该克制：

- 不急着做全部资源类型；
- 不急着完全自动化；
- 不急着追求 100% 准确；
- 先把目录树、正文挂载、人工审核、草稿保存这条链路打穿。

如果能让系统稳定完成 50% - 80% 的重复工作，对出版社和运营来说已经非常有价值。

AI 在这里不是魔法按钮，而是一个结构修复器、内容理解器和效率放大器。真正可靠的系统，还是要靠清晰的数据结构、可追踪的状态、人工审核和稳定的异步任务一点点堆出来。
