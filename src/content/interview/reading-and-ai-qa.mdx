# 从点读到 AI 问答：教育内容交互系统复盘

这篇是我对“点读 + AI 问答”这一整套教育内容交互能力的复盘。

点读解决的是“用户看到内容后，如何立刻听懂”；AI 问答解决的是“用户产生疑问后，如何基于书本内容得到解释”。它们看起来是两个功能，但底层都围绕同一件事：**把出版社已有的数字内容，变成可以被用户实时交互的学习资源**。

## 业务场景

出版社手里有大量教材、教辅、课本 PDF 和配套资源。传统使用方式通常是：

- 用户打开 PDF 或电子书阅读；
- 遇到不认识的句子，需要自己查；
- 听读、朗读、解释、练习题基本是割裂的；
- B 端运营人员需要手动维护大量资源。

所以点读和 AI 问答的目标不是“炫技”，而是提升内容资源的使用效率：

```
用户点击一句话
  → 系统知道点击的是哪段文本
  → 立刻合成或播放音频
  → 用户继续追问
  → AI 基于书本内容回答
```

这个体验一旦打通，PDF 就不再只是一个静态文件，而是变成了可以交互的学习界面。

## 整体链路

我更倾向把它拆成三层：

```
内容解析层
  PDF / 文档 / 教材资源
  → MinerU / OCR / 文本块解析
  → 页面、坐标、文本、结构化数据

交互能力层
  点读热区
  → TTS 合成
  → 音频播放
  → AI 问答检索

产品使用层
  小程序 / WebView
  → 用户点击、播放、提问
  → 后端异步任务、状态轮询、资源回写
```

这里最重要的是第一层。只要内容解析层不稳定，后面的点读、问答、TTS、资源生成都会跟着飘。

## 点读系统的核心难点

### 1. PDF 不是 HTML

HTML 里的文本天然有 DOM 结构，可以直接绑定点击事件。但 PDF 更像是一张排版后的“纸”，文本只是被绘制在某个坐标上。

所以点读的关键不是播放音频，而是：

> 用户点了页面上的某个位置，系统如何知道它对应哪一句话？

实现上需要提前解析出文本块坐标：

```json
{
  "page_no": 12,
  "text": "春天来了，小草从土里探出头来。",
  "bbox": [10.2, 22.4, 78.6, 26.1]
}
```

前端渲染 PDF 时，再叠加一层透明热区：

```
PDF 页面
  ↓
透明 bbox 覆盖层
  ↓
点击命中某个文本块
  ↓
调用 TTS / 播放缓存音频
```

### 2. 坐标必须按比例处理

不同设备、不同 DPR、不同缩放比例下，PDF 页面实际渲染尺寸都不一样。如果后端保存的是像素坐标，前端很容易对不上。

更稳的做法是保存百分比坐标：

```typescript
const style = {
  left: `${left}%`,
  top: `${top}%`,
  width: `${right - left}%`,
  height: `${bottom - top}%`,
};
```

这样无论页面被渲染成 375px 宽，还是 1024px 宽，热区都能跟着页面等比缩放。

### 3. TTS 不能阻塞用户

如果每次点击都等完整音频合成完再播放，用户会明显感觉慢。这里有两个优化方向：

- 短文本走实时合成，尽量降低首包时间；
- 高频内容提前生成音频，点击时直接播放 OSS 地址。

这其实也是“实时能力”和“离线资源生产”的取舍：

| 方式 | 优点 | 缺点 |
|------|------|------|
| 实时 TTS | 不需要提前生成，灵活 | 首次播放有等待 |
| 预生成音频 | 播放快，体验稳定 | 需要任务队列和存储成本 |

在教育内容里，我更倾向核心资源预生成，临时内容实时生成。这样体验和成本都比较均衡。

## AI 问答的核心难点

AI 问答不是把用户问题直接丢给大模型就完事了。对出版社内容来说，最关键的是：

> 回答必须尽量基于当前书本内容，不能自由发挥。

所以整体链路更像 RAG：

```
书本内容
  → 文本清洗
  → 切片
  → 向量化
  → 向量库

用户提问
  → 问题向量化
  → 检索相关片段
  → 拼接上下文
  → LLM 生成回答
```

### 为什么不能只靠 Prompt

Prompt 可以约束模型“请基于材料回答”，但如果材料没有被正确检索出来，模型还是会缺上下文。

所以问答质量取决于三件事：

- **内容切片**：切得太碎会丢上下文，切得太大检索不准；
- **检索召回**：用户问法和原文表达不一致时，能不能找回来；
- **回答约束**：模型是否严格基于材料，不乱编。

### 一个更稳的 Prompt 思路

```text
你是教材内容问答助手。

规则：
1. 只能基于提供的上下文回答。
2. 如果上下文没有答案，要明确说“当前材料中没有找到”。
3. 回答要适合学生理解，避免编造教材外信息。
4. 如果问题适合分步骤解释，请分点说明。
```

这里最重要的是第 2 条。教育内容宁可答得保守，也不能看起来很流畅但实际是错的。

## 前后端协作方式

这类功能前端不能只当 UI 写，后端也不能只给一个接口就结束。比较合理的协作边界是：

### 前端负责

- PDF 渲染；
- 热区覆盖层；
- 点击命中与交互反馈；
- 音频播放状态；
- AI 问答消息流式渲染；
- 用户正在处理、失败重试、空状态等体验。

### 后端负责

- PDF 解析；
- 文本块、页码、坐标结构化；
- TTS 合成与 OSS 上传；
- 异步任务状态；
- 向量化与检索；
- LLM 调用与安全约束。

中间必须有稳定的数据结构，比如：

```typescript
interface ReadingBlock {
  id: string;
  pageNo: number;
  text: string;
  bbox: [number, number, number, number];
  audioUrl?: string;
}
```

这类中间结构越稳定，功能就越容易继续扩展。后面无论做跟读、朗读评分、错题讲解，都会复用这套基础数据。

## 踩过的坑

### 1. 先写交互，后补数据结构，会很痛

点读功能最容易一开始就陷入页面效果，比如翻页、点击、播放动画。但真正决定成败的是数据结构：页码、坐标、文本块、音频 URL、解析状态。

如果这些字段一开始没设计清楚，后面每加一个能力都要返工。

### 2. 轮询状态必须能解释失败

异步任务不能只返回：

```json
{ "status": "processing" }
```

因为用户看到的永远是“处理中”，研发看到的也是“处理中”，问题被藏起来了。

更好的状态应该包含：

```json
{
  "status": "processing",
  "total": 58,
  "completed": 12,
  "failed": 1,
  "current": "正在合成第 13 段音频"
}
```

尤其是 TTS、OCR、LLM 这种外部服务调用，失败不可避免。状态设计必须把失败暴露出来。

### 3. AI 能增强体验，但不能替代业务规则

AI 问答可以解释内容，可以补充说明，可以生成更自然的回答。但资源归属、教材结构、章节边界、内容审核这些东西，还是必须落到业务规则和人工确认上。

这也是我现在越来越认同的一个原则：

> AI 适合做“理解和生成”，系统必须负责“结构和确定性”。

## 我对这类产品的理解

点读和 AI 问答本质上不是两个孤立功能，而是教育内容数字化之后的两个入口：

- 点读让用户从“看内容”变成“听内容”；
- AI 问答让用户从“读内容”变成“问内容”；
- TTS、OCR、RAG、LLM 这些技术最终都是为了让内容更容易被使用。

对 B 端出版社来说，真正有价值的不是“我们接了一个大模型”，而是：

```
已有内容资产
  → 被解析
  → 被结构化
  → 被生成音频
  → 被问答检索
  → 被用户消费
  → 被后台继续管理
```

当这个闭环跑起来，出版社的 PDF、教材、教辅就不再只是文件，而是可以持续生产增值服务的内容资产。

这也是我觉得这个方向值得继续做的原因：它不是单点功能，而是内容平台能力的一部分。
