# 微信第三方平台：一份代码发布到 N 家小程序

我接手了公司一套挺特别的发布体系：我们不是发一个独立小程序，而是用微信「第三方平台（服务商 / 代开发）」模式，把**同一份小程序代码**，发布到十几家出版社各自的小程序上。

这套机制是整个 C 端能不能发版的命门，也是最容易绕晕的部分。这篇把它的本质、两套 token 的关系、完整发布状态机、以及「同一份代码怎么区分是哪家」讲清楚。

## 本质：一套代码，N 个小程序

先说清楚我们**不是**一个独立小程序，而是微信开放平台的「第三方平台 / 服务商代开发」模式：

```
一份模板代码 (jilan-wx-app)
   └── 上传成「模板库」拿到 template_id
        └── 批量推送 + 给每家注入专属 ext.json
             └── 部署到 N 家出版社各自授权的小程序
                  (德显龙 / 龙图 / 起点 …)
```

每家出版社 = 一个独立 appId + 一条授权记录。跑的是**同一份代码**，靠 `ext.json` 注入的 `publisherId` 区分身份。

这里有个最容易误解的点，我也踩过：**微信开发者工具里「上传体验版」只能用于自己本地自测。** 想让那十几家出版社的线上用户更新，必须走第三方平台 API 这一整套流程，跟点一下「上传」完全是两回事。

## 两套 token 怎么来的

整套体系最绕的就是 token，有两层，先理清关系：

```
微信每 10 分钟推 component_verify_ticket
        │  (webhook 收到, 存库)
        ▼
  换 component_access_token  (服务商级, 2h 过期)
        │
        ├──→ 调「模板库」接口 (拉草稿 / 加模板)
        │
        └──→ 换 authorizer_access_token  (某一家小程序级, 2h 过期)
                    │
                    └──→ 代这家小程序: commit / 提审 / 发布 / 改域名
```

- **component（服务商）**：我们公司在微信开放平台注册的代开发主体，全局只有一个。
- **component_verify_ticket**：微信每 10 分钟主动推给我们的票据，是换服务商 token 的原料。如果这个票据断了（比如 webhook 挂了），**所有家的发布全瘫**。
- **component_access_token**：服务商级 token，用来调模板库、以及去换下一层的授权 token。
- **authorizer_access_token**：代某一家小程序操作的 token，commit / 审核 / 发布 / 改域名都用它。

两层 token 都 2 小时过期，代码全程自动续期——调用前取「有效 token」，过期了透明刷新重试一次。基本不用手动管。

唯一要警惕的信号：某家报 `AUTHORIZER_REFRESH_TOKEN_MISSING`，说明这家的授权坏了或被取消，需要出版社**重新扫码授权**给我们服务商。

## 完整发布链路（状态机）

每家小程序的一次发版，是一个明确的状态机：

```
DRAFT ──→ CODE_COMMITTED ──→ SUBMITTED ──→ APPROVED ──→ PUBLISHED
  建版本      commit 代码        提交审核      微信过审       release
                                              │
                                              └──→ REJECTED (驳回)
```

逐步拆解：

**第 0 步｜模板库（全局一次，所有家共享）**。代码改动要先进模板库：开发者工具把 `jilan-wx-app` 代码上传成「草稿」→ B 端超管拉草稿列表 → 把草稿加入模板库，拿到新 `template_id`。这个 `template_id` 是后面所有出版社共用的同一份代码。

**第 1 步｜建版本（每家一条）**。选一个出版社授权 + 绑定 `template_id` + 填版本号，状态 `DRAFT`。唯一键 `(authorizationId, version)` 防重复。

**第 2 步｜提审（核心）**。严格按顺序做 4 件事，任何一步失败都中断：

```
1. 同步服务器域名到这家小程序  (modify_domain)
2. 同步隐私声明              (setprivacysetting)
3. commit 代码 + 这家专属 ext.json → 状态 CODE_COMMITTED
4. 提交微信审核                → 状态 SUBMITTED, 拿 auditId
```

**第 3 步｜审核状态回写**。两条路：微信审核完主动推 webhook（主），把版本改 `APPROVED` / `REJECTED`；或主动轮询审核状态接口（兜底）。

**第 4 步｜发布**。仅 `APPROVED` 可发，调 `release` → `PUBLISHED`，C 端用户生效。

## ext.json：同一份代码区分身份的魔法

第 2 步里 commit 代码时，会给每家注入一份专属的 `ext.json`：

```json
{
  "extEnable": true,
  "extAppid": "<这家的 authorizerAppId>",
  "ext": {
    "publisherId": "9002",
    "authorizationId": "..."
  }
}
```

这就是「一份代码区分出版社」的核心。同一份模板代码，部署到德显龙的小程序里 `publisherId` 是 9002，部署到另一家就是另一个值。代码运行时读这个值，就知道自己「现在是哪家」。

## C 端闭环：小程序怎么知道自己是哪家

发布是注入身份，运行时是读出身份，闭环是这样合上的：

```
小程序启动
  → wx.getAccountInfoSync() 拿到自己的 appId
  → 请求带 header  X-App-Id: <appId>
  → next 后端 resolvePublisherByAppId(appId)
       (按 publishers.mini_program_app_id 反查)
  → 拿到 publisher → 返回这家的书 / 资源 / 能力开关
```

注意这里用的是 `appId` 反查 `publisher`，而不是直接信任 `ext.json` 里的 `publisherId`——后端以 appId 为准更可靠。

也正因为这个闭环，做「灰度只给一家开新功能」特别简单：用出版社级开关。比如新做的「资源层级树」只给德显龙开，后端判断 `publisherId` 在不在白名单里，别家接口直接返「不开启」走老逻辑，**零感知、零风险**。已经在生产验证过：德显龙出树，其它家平铺，互不影响。

## 哪些能批量，哪些只能逐家

这是接手时最该先搞清的能力边界：

| 操作 | 批量？ | 说明 |
|------|--------|------|
| 上传模板库 | 全局一次 | 所有家共享一个 template_id |
| 建版本 | ❌ 一家一条 | |
| 域名同步 | ✅ | 改全局域名时自动同步所有已授权的家 |
| 提审 | ❌ 一家一条 | 目前要逐家点 |
| 发布 | ✅ | 勾选批量，串行循环，单家失败不阻断其余 |

现状是「提审逐家、发布可批量」。如果将来要「一键全量提审」，照着批量发布的串行模式加一个就行。

## webhook 回调入口

出问题先查这三个入口：

| 路径 | 收什么 |
|------|--------|
| `/api/wechat-third-party/events` | `component_verify_ticket` / 授权变更（授权 / 取消授权） |
| `/api/wechat-third-party/messages` | 小程序消息 + 审核事件 |
| `/api/mini-program-versions/wechat-callback` | 审核结果（成功 / 失败 / 延后） |

所有事件都落一张审核记录表（带来源标记 `WECHAT` / `SYSTEM`），排查审核问题去翻这张表的原始 payload。

## 踩过的坑

**1. commit 后不能马上提审。** commit 完微信要跑一个「隐私接口检查」任务，没跑完就提审会报「隐私接口检查任务未完成」。代码捕获了这个错，提示稍后再点——而且因为状态已经是 `CODE_COMMITTED`，重试时会跳过 commit 直接提审，不会重复 commit。

**2. 本地别改 appId 去模拟别家。** 我踩过：在开发者工具里改 `project.config.json` 的 appid 去模拟另一家，会清空登录态 storage，强制走 `wx.login`，然后 next 调微信 `jscode2session` 时被**微信 IP 白名单**拦（报 `invalid ip not in whitelist`）。正确做法是配一个 debug token 绕过 `wx.login`，或者直接 curl 后端接口带 `x-app-id` header 来验证别家。

**3. 票据断了是最高优先级故障。** `component_verify_ticket` 是整套体系的根，微信每 10 分钟推一次。排查发布全瘫时，第一件事是看票据表的更新时间是不是 10 分钟内的，断了就是 webhook 出了问题，所有发布动作都会失败。

**4. 模板库有上限。** 模板库容量上限约 50 个，需要定期删旧模板维护，不然加不进新草稿。

## 我对这套设计的理解

这套机制本质上是一个**多租户发布系统**：一份代码，通过「模板库 + 配置注入」分发到 N 个租户，每个租户有独立身份、独立审核、独立发布节奏。

它最巧妙的地方是把「代码」和「身份」彻底分开：

- 代码是共享的（一个 template_id），改一次所有家都能用上；
- 身份是注入的（每家一份 ext.json），运行时读出来决定「我是谁、该返哪家的数据」。

这种「共享逻辑 + 注入配置」的思路，其实不止小程序——任何 SaaS 多租户场景都适用。难点不在某个 API 怎么调，而在于把两层 token 的生命周期、发布状态机、灰度隔离这几件事理清楚，并且每一步都能从日志 / 状态表里查到「现在卡在哪、为什么卡」。

对接手者来说，真正要吃透的不是代码，而是这条链路上**每一个环节出问题时去哪儿查**。我把它写下来，一半是给自己备忘，一半是怕哪天这套东西又只剩我一个人懂。
