# TTS 多供应商路由与语音引擎选型

我们的教育产品有三条链路都要把文字变成语音：点读（逐句朗读）、拆书（整章音频）、AI 转语音（运营手动合成）。一开始只接了阿里云 CosyVoice，后来陆续接了通义千问 Qwen-TTS、调研了火山豆包语音。

这篇分两部分：一是**怎么调研选型**（不同引擎的能力 / 计费 / SSML 取舍），二是**怎么把多供应商做成可插拔的路由**，让接第三家时调用方零改动。

## 先说结论

1. 「精确控制」和「音色自然度」是当前 TTS 的一对矛盾，按场景选型，不要指望一家通吃。
2. 新一代大模型 TTS 普遍在**淡化 SSML**，改用模型自身韵律 + 自然语言指令。但多音字注音 `<phoneme>` 各家都保留。
3. 我们点读场景对停顿标签依赖很低（逐句切片本身就实现了停顿），真正离不开的只有多音字注音。
4. 架构上用 `TtsProvider` 抽象 + `tts_provider` 配置表 + 按音色路由，接第三家供应商只要加一个实现类 + 灌一行配置，三条调用链路完全无感。

## 一对核心矛盾：精确控制 vs 自然度

做教育内容的 TTS，你会很快撞到这个矛盾：

- 要**精确控制**——手动指定停顿、给多音字注音、按位读数字（「读 1234 为一二三四」而不是「一千二百三十四」）→ 得用支持完整 SSML 的「老一代」引擎。
- 要**自然好听 + 有情感**——→ 得用「新一代大模型」引擎，但要放弃 SSML 硬标签，改用切片 + 标点 + 自然语言指令。

教材点读偏前者（准确性第一），展示类、有情感的朗读偏后者。一个引擎很难两头都占。

## 为什么新一代大模型在淡化 SSML

这点我专门查证过，因为一开始我也奇怪：怎么越「先进」的引擎反而 SSML 支持越差？

原因是技术路线变了：

- **老模型不懂语义**，靠 SSML 这种外部标签来「指挥」它在哪停顿、用什么语调；
- **新大模型能理解上下文**，自己就能决定韵律，于是改用自然语言指令（比如 `instructions: "用轻快的语气，稍微快一点"`）来控制，而不是硬标签。

但有个例外：`<phoneme>`（多音字注音）几乎各家都保留。因为多音字到底读哪个音，模型自己猜不准——「行」读 xíng 还是 háng，得靠外部明确告诉它。这对中文教材尤其关键。

一个最坑的细节：千问 Qwen-TTS **完全不支持 SSML**，但你把 SSML 标签塞进 text 里，它**不报错，标签被静默忽略**——HTTP 200 成功，但语义层无效。这种「假成功」最难排查，因为日志看起来一切正常，只有听音频才发现标签没生效。

## 三家能力对比

> ⚠️ 计费单位口径不同：阿里 1 汉字 = 2 字符，火山 1 汉字 = 1 字符，比价务必先换算。

| 维度 | 阿里 CosyVoice | 千问 Qwen-TTS | 火山豆包（分老/新两条线） |
|------|---------------|---------------|--------------------------|
| 数字语速 speech_rate | ✅ [0.5, 2.0] | ❌（靠 instruction 文案） | ✅ |
| SSML `<break>` 停顿 | ✅ | ❌ | 老线 ✅ / 新线 ❌ |
| SSML `<phoneme>` 多音字 | ✅ | ❌ | ✅（各线都有） |
| 自然语言指令（语气/情感） | ❌ | ✅ 仅 instruct 版 | ✅ 新线 |
| 字/音素时间戳 | ✅ | ❌ | ✅ |
| 单次文本上限 | 20000 字符 | 512 token | 约 300~1000 字 |

我们实际接入：CosyVoice 用 `cosyvoice-v3-flash`；千问用 `qwen3-tts-flash`（默认），带指令时切 `qwen3-tts-instruct-flash`。

火山豆包要特别注意它有**两条独立产品线**，很容易混：

- **新线（豆包大模型 2.0，`_uranus_bigtts` 音色）**：音色最自然、有情感指令、能朗读 Latex 公式，但 SSML 只剩 `<phoneme>`。
- **老线（精品长文本，`_mars_bigtts` / `_moon_bigtts`）**：完整 SSML（含停顿），音色传统些，但便宜。

## 计费这块的坑

统一折算成「每万汉字（后付费）」之后才好比：

| 引擎 / 线 | 每万汉字 | SSML | 备注 |
|-----------|---------|------|------|
| 火山 精品长文本 | ≈1 元 | ✅ 完整 | 性价比最高 + 有停顿 |
| 千问 qwen3-tts-flash | ≈1.6~2 元 | ❌ | 新音色，无 SSML |
| 火山 豆包大模型 2.0（新音色） | ≈3 元 | ⚠️ 仅 phoneme | 音色最自然，贵 |
| 火山 大模型语音合成（老大模型线） | ≈5 元 | ✅ | 偏贵 |

有两个成本放大点必须盯：

1. **点读是「逐句 × 多音色」全量预合成**，预合成音色数是成本乘数。每多标一个预合成音色，整本点读成本线性翻倍。我们控制在 3 个音色。
2. **换音色重新生成 = 重新全量计费**，所以运营换音色要谨慎。好在合成层做了幂等：已完成的句子/章节会跳过，重试不重复计费。

## 工程：可插拔的供应商路由

选型可以慢慢调研，但代码不能每接一家就改一遍调用方。我的做法是抽一个统一接口 `TtsProvider`，把「用谁合成」这件事彻底藏起来。

```python
class TtsProvider(ABC):
    """所有供应商实现这个接口。调用方（点读/拆书/通用任务）只依赖它。"""

    @abstractmethod
    def synthesize_sync(
        self, text: str, voice: str,
        model: Optional[str] = None,
        speed: Optional[float] = None,        # 数字语速, 不支持的供应商忽略
        instructions: Optional[str] = None,   # 自然语言指令, 不支持的忽略
    ) -> bytes:
        """同步合成, 返回 MP3 bytes"""
        ...
```

接口设计上有个关键取舍：**非流式（`synthesize_sync`）所有供应商必须实现，流式各家协议差异太大（阿里 WebSocket callback、火山、讯飞各不同），基类默认抛 `NotImplementedError`，支持的供应商自己覆写。** 这样不会因为某家不支持流式就卡住整个抽象。

参数也体现了「能力差异」：`speed` 只对支持数字语速的供应商（如 CosyVoice 的 `speech_rate`）生效，千问非实时没有数字语速会忽略；`instructions` 只有千问 instruct 版认。接口统一收下，由各实现自己决定理不理。

配置全部抽到一张 `tts_provider` 表，不再散在 `.env`：

```python
class TtsProvider(Base):
    __tablename__ = "tts_provider"
    provider_key   = Column(String(60))   # aliyun_cosyvoice / qwen_tts / volcengine
    provider_type  = Column(String(40))   # 决定用哪个实现类
    api_key        = Column(String(500))  # 直接存表, 前端遮罩显示
    default_model  = Column(String(100))
    params_json    = Column(JSON)         # max_concurrent / requests_per_second / timeout
    is_default     = Column(Integer)      # 不指定供应商时用它
    enabled        = Column(Integer)
```

好处是运营在后台就能切供应商、改限流参数、换 key，改完即生效，不用改代码、不用重启。

## 按音色路由：音色就是供应商的载体

这里有个我觉得挺顺的设计点：**到底用哪家合成，不需要业务方显式指定，靠音色就能推断出来。**

因为音色和供应商是绑死的——CosyVoice 的「龙珊珊」和千问的「Cherry」是两家的东西，不可能混用。所以 `voices` 表里存了每个音色属于哪个 `provider`，路由器拿到 `voice_id` 反查就知道走哪家：

```python
class TtsRouter:
    async def resolve_for_voice(self, voice_id: str) -> TtsProvider:
        """按音色路由到它所属的供应商。查不到就回落默认供应商。"""
        provider_key = await self._provider_key_for_voice(voice_id)  # 查 voices.provider
        return await self.resolve(provider_key)
```

调用方（点读、拆书、AI 转语音）只管「我要用这个音色合成这段话」，完全不知道背后是阿里还是千问。

路由器还做了两件工程上的事：

- **60s 内存缓存**：配置改完最多 60s 生效，绝大多数调用命中缓存不查库。
- **sync / async 双版本**：FastAPI 主流程走 async，Celery 同步任务链走 sync 版（独立线程 + 独立 engine 跑临时事件循环）。

为什么 sync 版要这么折腾独立 engine？这是踩出来的——Celery worker 每个任务跑在一个临时事件循环里，如果复用主引擎的连接池，第二个任务会借到上一个已关闭 loop 的池化连接，直接报 `got Future attached to a different loop`。所以 worker 场景必须用「跨 loop 安全」的一次性 session。

## 接第三家要改什么

这是检验抽象好不好的标准。接火山豆包时，改动清单是：

1. 写一个 `VolcengineProvider(TtsProvider)` 实现类（封装它的鉴权 + HTTP 协议）；
2. 在路由器的 `_PROVIDER_IMPLS` 字典里注册一行 `"volcengine": VolcengineProvider`；
3. `tts_provider` 表里灌一行配置；
4. `voices` 表灌入豆包音色，诚实标好 `ssml_support` / `instruct_support` / `speed_support`。

**调用方（三条链路）一行都不用动。** 这就是抽象到位的样子——变化被关在 provider 实现和配置表里，没有泄漏到业务。

## 踩过的坑

**1. 千问会忽略外部传入的非 qwen 模型名。** 我们的合成任务默认带 `model=cosyvoice-v3-flash`（历史默认值）。这个值传给千问 provider，dashscope 拿非法模型名拼 URL 直接报 `url error`。所以 provider 内部要做防御：只认 `qwen` 开头的模型名，其它一律回落自己的默认模型。

```python
passed = (model or "").strip()
base_model = passed if passed.startswith("qwen") else self.config.default_model
```

**2. 音色能力字段一开始有脏数据。** CosyVoice 音色被错标了 `instruct_support=1`，千问音色 `ssml_support` 该是 0 却没标。前端控件是按这些字段驱动显隐的（支持指令才显示指令输入框），标错就会给用户显示一个根本不生效的控件。后来专门修了一轮脏数据。

**3. 前端控件不要写死。** 「语速滑块」「语气指令框」这些控件显不显示，全部由 `voices` 表的 `speed_support` / `instruct_support` 字段驱动。换供应商、加音色时，前端自动适配，不用改代码。

**4. TTS 失败要可观测。** AI 转语音、点读合成失败都接入了 `ai_error_log`，前端有专门的「错误日志」页能查。外部服务调用失败是常态，把失败显式记下来比「卡在处理中」强太多。

## 我的选型建议

| 场景 | 推荐 | 理由 |
|------|------|------|
| 教材点读（要多音字、低成本、批量） | 火山精品长文本 / 阿里 CosyVoice | 完整 SSML + 便宜；点读切片本身已实现停顿 |
| 有情感的展示音色 | 千问 instruct / 火山豆包 2.0 | 自然度高、有情感指令 |
| 公式朗读（理科教材） | 火山豆包 2.0 | 独有 Latex 公式朗读 |
| 方言 / 外语 | 千问 / 火山 | 音色多，看具体试听 |

最务实的结论：**火山「精品长文本」线 ≈ 便宜（1 元/万汉字）+ 完整 SSML（有停顿）+ 多音字注音**，是点读 / 拆书的潜在最优解。

但比选型更重要的是：因为前面那套 `TtsProvider` 抽象 + 配置表 + 按音色路由，选型这件事现在是**可逆的**。哪家涨价了、哪家限流了、哪家出了更好的音色，我都能在不动业务代码的前提下切过去。对一个还在快速变化的领域（大模型 TTS 半年一个新东西），这种「不被供应商绑死」的能力，比当下选对哪一家更值钱。
