# React 长文本流式渲染性能优化

> 面试高频题：大模型生成 5000+ 字的超长 Markdown，如何保证页面不卡顿？

## 🎯 问题场景

### 性能瓶颈

假设大模型已经生成了 5000 字的超长 Markdown 文章，并且还在继续生成。

**如果每 16ms 就把这 5000+ 字的完整字符串塞给 Markdown 组件**：

```typescript
// ❌ 性能灾难
function ChatMessage({ content }: { content: string }) {
  return <MarkdownRenderer content={content} />;  // 5000+ 字
}

// 每 16ms 触发一次：
// 1. Markdown 解析（5000+ 字）
// 2. 生成虚拟 DOM
// 3. Diff 算法对比
// 4. 渲染到真实 DOM
// → 主线程卡死，掉帧严重
```

**问题**：
- Markdown 解析耗时（正则、AST）
- 虚拟 DOM 生成耗时（5000+ 个节点）
- Diff 算法耗时（对比 5000+ 个节点）
- DOM 操作耗时（重排重绘）

---

## 💡 架构为什么不卡顿？

### 核心策略：事件合并 + 增量更新


#### 1. 事件级别的合并（Store 层）

```typescript
// multiSessionStore.ts
handleSSEEvent: (sessionId: string, event: StreamEvent) => {
  // 结束事件立即处理
  if (event.type === StreamEventType.COMPLETE || event.type === StreamEventType.ERROR) {
    const bufferedEvents = flushEventBuffer(sessionId);
    if (bufferedEvents.length > 0) {
      addBatchStreamEvents(sessionId, streamingMessageId, bufferedEvents);
    }
    completeAssistantMessage(sessionId, streamingMessageId, event);
    return;
  }

  // 其他事件：使用批量处理优化
  scheduleEventBatch(sessionId, event, () => {
    const events = flushEventBuffer(sessionId);
    if (events.length > 0) {
      addBatchStreamEvents(sessionId, streamingMessageId, events);
    }
  });
}
```

**关键点**：
- 每收到一个 SSE chunk，不是立即更新 React 状态
- 而是先 push 到 Buffer
- RAF 在每一帧 flush Buffer 中的所有事件

#### 2. 消息级别的合并（Processor 层）

```typescript
// streamEventProcessor.ts
export const processStreamEvents = (events: StreamEvent[]): ProcessedEvent[] => {
  const processed: ProcessedEvent[] = [];

  // 处理 message 事件：合并连续的消息
  let i = 0;
  while (i < sortedEvents.length) {
    const event = sortedEvents[i];

    switch (event.type) {
      case StreamEventType.MESSAGE:
        // ⭐ 关键：合并连续的 message 事件
        const messageEvents = [event];
        let j = i + 1;
        while (j < sortedEvents.length && sortedEvents[j].type === StreamEventType.MESSAGE) {
          messageEvents.push(sortedEvents[j]);
          j++;
        }
        
        // 合并所有内容为一个字符串
        const combinedContent = messageEvents.map((e) => e.content || "").join("").trim();
        
        // 只生成一个 ProcessedEvent
        processed.push({ 
          type: "message", 
          content: combinedContent,  // 完整的 5000+ 字
          originalEvents: messageEvents 
        });
        
        i = j;
        break;
    }
  }

  return processed;
};
```

**关键点**：
- 将多个 MESSAGE 事件合并为一个 ProcessedEvent
- 只有一个 `<MarkdownRenderer>` 组件
- 内容是累积的完整字符串

#### 3. 渲染层的 RAF 优化

```typescript
// StreamEventRenderer.tsx
export const StreamEventRenderer: React.FC<StreamEventRendererProps> = React.memo(
  ({ processedEvents, isStreaming, isPaused }) => {
    // RAF 优化高频更新
    const [optimizedEvents, setOptimizedEvents] = React.useState(processedEvents);
    const rafIdRef = React.useRef<number | null>(null);

    React.useEffect(() => {
      if (rafIdRef.current) cancelAnimationFrame(rafIdRef.current);

      rafIdRef.current = requestAnimationFrame(() => {
        setOptimizedEvents(processedEvents);
        rafIdRef.current = null;
      });

      return () => {
        if (rafIdRef.current) cancelAnimationFrame(rafIdRef.current);
      };
    }, [processedEvents]);

    return (
      <div className="space-y-2">
        {optimizedEvents.map((item, i) => {
          const key = `${item.originalEvents.map((e) => e.id).join("_")}_${i}`;

          switch (item.type) {
            case "message":
              // ⭐ 只有一个 MarkdownRenderer 组件
              return <MarkdownRenderer key={key} content={item.content} />;
            // ...
          }
        })}
      </div>
    );
  }
);
```

**关键点**：
- 使用 RAF 限制更新频率（60fps）
- 使用 React.memo 避免不必要的重渲染

## 🔥 为什么这样不卡顿？

### 数据流分析

```
SSE 流（100 chunk/s）
  ↓
每个 chunk 约 50 字
  ↓
scheduleEventBatch (RAF 批量)
  ↓
每帧（16ms）收集约 1-2 个 chunk
  ↓
flushEventBuffer → 返回 1-2 个 MESSAGE 事件
  ↓
processStreamEvents → 合并为 1 个 ProcessedEvent
  ↓
StreamEventRenderer (RAF 优化)
  ↓
MarkdownRenderer 渲染 5000+ 字
  ↓
React Diff 算法
  ↓
DOM 更新
```

### 关键优化点

#### 1. 事件合并减少了 React 更新次数

**没有合并**：
```
100 个 chunk → 100 次 setState → 100 次 React 渲染
```

**有合并（RAF）**：
```
100 个 chunk → 6 次 setState (60fps) → 6 次 React 渲染
```

**减少了 94% 的渲染次数！**

#### 2. 单个 Markdown 组件 vs 多个组件

**错误做法**：
```typescript
// ❌ 每个 chunk 一个组件
{events.map(event => (
  <MarkdownRenderer key={event.id} content={event.content} />
))}

// 5000 字 = 100 个 chunk = 100 个 MarkdownRenderer
// Diff 算法要对比 100 个组件
```

**正确做法**：
```typescript
// ✅ 合并为一个组件
const combinedContent = events.map(e => e.content).join('');
<MarkdownRenderer content={combinedContent} />

// 5000 字 = 1 个 MarkdownRenderer
// Diff 算法只对比 1 个组件
```

#### 3. React Diff 算法的优化

```typescript
// React Diff 算法（简化版）
function reconcileChildren(current, workInProgress, nextChildren) {
  // 1. 检查 key 是否相同
  if (current.key === workInProgress.key) {
    // 2. 检查 type 是否相同
    if (current.type === workInProgress.type) {
      // 3. 检查 props 是否相同
      if (current.props.content === workInProgress.props.content) {
        // ⭐ 完全相同，复用旧节点，不重新渲染
        return current;
      } else {
        // ⭐ 只有 content 变了，更新文本节点
        updateTextContent(current, workInProgress.props.content);
      }
    }
  }
}
```

**关键点**：
- 因为 key 和 type 都相同，React 只需要更新文本内容
- 不需要重新创建 DOM 节点
- 不需要重新解析 Markdown（如果 Markdown 组件做了缓存）

## 🚀 进一步优化策略

### 1. Markdown 组件内部优化

```typescript
import React, { useMemo } from 'react';
import ReactMarkdown from 'react-markdown';
import remarkGfm from 'remark-gfm';
import rehypeHighlight from 'rehype-highlight';

const MarkdownRenderer = React.memo(({ content }: { content: string }) => {
  // ⭐ 使用 useMemo 缓存解析结果
  const parsedContent = useMemo(() => {
    return content;  // 实际上 ReactMarkdown 内部会缓存
  }, [content]);

  return (
    <ReactMarkdown
      remarkPlugins={[remarkGfm]}
      rehypePlugins={[rehypeHighlight]}
    >
      {parsedContent}
    </ReactMarkdown>
  );
}, (prev, next) => {
  // ⭐ 自定义比较函数：只有 content 变化才重新渲染
  return prev.content === next.content;
});

export default MarkdownRenderer;
```

### 2. 虚拟滚动（超长文本）

如果文本超过 10000 字，可以使用虚拟滚动：

```typescript
import { useVirtualizer } from '@tanstack/react-virtual';

function LongTextRenderer({ content }: { content: string }) {
  const parentRef = useRef<HTMLDivElement>(null);
  
  // 将文本按段落分割
  const paragraphs = useMemo(() => {
    return content.split('\n\n');
  }, [content]);
  
  const virtualizer = useVirtualizer({
    count: paragraphs.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 100,  // 每段约 100px
  });
  
  return (
    <div ref={parentRef} style={{ height: '600px', overflow: 'auto' }}>
      <div style={{ height: `${virtualizer.getTotalSize()}px`, position: 'relative' }}>
        {virtualizer.getVirtualItems().map(virtualItem => (
          <div
            key={virtualItem.index}
            style={{
              position: 'absolute',
              top: 0,
              left: 0,
              width: '100%',
              transform: `translateY(${virtualItem.start}px)`,
            }}
          >
            <MarkdownRenderer content={paragraphs[virtualItem.index]} />
          </div>
        ))}
      </div>
    </div>
  );
}
```

### 3. 增量渲染（分段加载）

```typescript
function IncrementalMarkdownRenderer({ content }: { content: string }) {
  const [visibleContent, setVisibleContent] = useState('');
  const [renderIndex, setRenderIndex] = useState(0);
  
  useEffect(() => {
    // 每次只渲染 1000 字
    const CHUNK_SIZE = 1000;
    
    if (renderIndex < content.length) {
      const nextChunk = content.slice(0, renderIndex + CHUNK_SIZE);
      
      requestIdleCallback(() => {
        setVisibleContent(nextChunk);
        setRenderIndex(renderIndex + CHUNK_SIZE);
      });
    }
  }, [content, renderIndex]);
  
  return <MarkdownRenderer content={visibleContent} />;
}
```

### 4. Web Worker 解析 Markdown

```typescript
// markdown-worker.ts
import { unified } from 'unified';
import remarkParse from 'remark-parse';
import remarkGfm from 'remark-gfm';
import remarkRehype from 'remark-rehype';
import rehypeStringify from 'rehype-stringify';

self.onmessage = async (e) => {
  const { content } = e.data;
  
  // 在 Worker 中解析 Markdown
  const result = await unified()
    .use(remarkParse)
    .use(remarkGfm)
    .use(remarkRehype)
    .use(rehypeStringify)
    .process(content);
  
  // 返回 HTML
  self.postMessage({ html: String(result) });
};
```

```typescript
// 主线程
function MarkdownRenderer({ content }: { content: string }) {
  const [html, setHtml] = useState('');
  const workerRef = useRef<Worker | null>(null);
  
  useEffect(() => {
    if (!workerRef.current) {
      workerRef.current = new Worker('/markdown-worker.js');
      workerRef.current.onmessage = (e) => {
        setHtml(e.data.html);
      };
    }
    
    workerRef.current.postMessage({ content });
  }, [content]);
  
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
}
```

### 5. 代码高亮懒加载

```typescript
import React, { Suspense, lazy } from 'react';

// 懒加载代码高亮组件
const CodeBlock = lazy(() => import('./CodeBlock'));

function MarkdownRenderer({ content }: { content: string }) {
  return (
    <ReactMarkdown
      components={{
        code: ({ node, inline, className, children, ...props }) => {
          if (inline) {
            return <code className={className} {...props}>{children}</code>;
          }
          
          return (
            <Suspense fallback={<pre><code>{children}</code></pre>}>
              <CodeBlock className={className} {...props}>
                {children}
              </CodeBlock>
            </Suspense>
          );
        },
      }}
    >
      {content}
    </ReactMarkdown>
  );
}
```

## 🎓 面试话术

### 问题：5000+ 字的 Markdown 如何保证不卡顿？

> "我的架构采用了**三层优化策略**：
>
> **第一层：事件级别的批量处理**。每收到一个 SSE chunk，不是立即更新 React 状态，而是先 push 到 Buffer。然后用 RAF 在每一帧 flush Buffer 中的所有事件。这样 100 个 chunk 只会触发 6 次 React 更新（60fps）。
>
> **第二层：消息级别的合并**。在 processStreamEvents 中，我会将连续的 MESSAGE 事件合并为一个 ProcessedEvent。这样 100 个 chunk 最终只生成 1 个 MarkdownRenderer 组件，而不是 100 个。React Diff 算法只需要对比 1 个组件，而不是 100 个。
>
> **第三层：组件级别的优化**。MarkdownRenderer 使用 React.memo 和自定义比较函数，只有 content 变化才重新渲染。同时使用 useMemo 缓存解析结果。
>
> 这三层优化叠加，即使是 5000+ 字的长文本，也能保持 60fps 的流畅渲染。"

### 问题：如果文本超过 10000 字呢？

> "如果文本超过 10000 字，我会引入**虚拟滚动**。将文本按段落分割，只渲染可见区域的段落。这样无论文本多长，DOM 节点数量都是固定的（约 20 个），性能不会随文本长度增长而下降。
>
> 另外，我还会考虑使用 **Web Worker** 将 Markdown 解析放到后台线程，避免阻塞主线程。"

### 问题：为什么不用 shouldComponentUpdate？

> "React.memo 是函数组件的 shouldComponentUpdate。我使用 React.memo 的自定义比较函数，只比较 content 是否变化。这比 shouldComponentUpdate 更简洁，也更符合函数式编程的思想。"

## 📚 总结

### 你的架构的核心优势

1. **事件批量处理（RAF）** - 减少 94% 的渲染次数
2. **消息合并** - 100 个组件变成 1 个组件
3. **React.memo** - 避免不必要的重渲染
4. **单一数据源** - 累积的完整字符串，而不是增量片段

### 为什么不卡顿？

- **减少了渲染次数**：100 次 → 6 次
- **减少了组件数量**：100 个 → 1 个
- **减少了 Diff 成本**：对比 100 个组件 → 对比 1 个组件
- **减少了 DOM 操作**：更新 100 个节点 → 更新 1 个节点

### 进一步优化方向

- 虚拟滚动（超长文本）
- Web Worker（后台解析）
- 增量渲染（分段加载）
- 代码高亮懒加载
