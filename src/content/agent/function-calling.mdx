# Function Calling 工具调用

## 我怎么理解 Function Calling

Function Calling 可以先理解成：**模型不一定直接回答，它也可以先告诉你的程序“该调哪个函数、参数是什么”。**

我一开始容易把它想得很神，其实落到代码里并不玄。模型不会真的去查天气、查数据库，它只是按工具定义生成一段结构化结果，真正执行工具的还是我们的应用代码。

```
普通对话：
  用户: "今天北京天气怎么样？"
  LLM:  "我无法获取实时天气信息..."  ← 只能靠训练数据

Function Calling：
  用户: "今天北京天气怎么样？"
  LLM:  { "function": "getWeather", "arguments": { "city": "北京" } }  ← 告诉你该调什么
  系统:  调用 getWeather("北京") → 返回 "晴，28°C"
  LLM:  "今天北京天气晴朗，气温 28°C，适合出门。"  ← 结合真实数据回答
```

如果用前端类比，LLM 有点像一个组件，Function Calling 像 `props.onAction`。组件自己不直接改外部世界，而是把“我想做什么”告诉外层，由外层真正去执行。

## 模型到底在干什么

回到 Transformer 那篇，模型本质上还是在预测下一个 token。Function Calling 并不是给模型开了一个隐藏开关，而是让模型在合适的时候学会输出特定格式的 JSON。

```
模型的内心独白（简化版）：

输入 tokens: [系统提示, 可用函数定义, 用户问题]
  ↓ Attention 计算
模型判断: "这个问题需要调用外部函数才能回答"
  ↓ 自回归生成
输出 tokens: {"function": "getWeather", "arguments": {"city": "北京"}}
```

所以我会把这件事拆成四步看：
1. 你在 system prompt 里告诉模型"你有哪些工具可以用"
2. 模型通过 Attention 理解用户意图 + 可用工具
3. 模型决定是直接回答还是调用工具
4. 如果调用工具，模型生成结构化的 JSON（函数名 + 参数）

这一点很重要：**模型不会真的执行函数。它只是生成“调用意图”，执行、校验、重试、兜底都在我们自己的代码里。**

## 实际开发里会走几步

```
┌─────────────┐
│  你的应用代码  │
└──────┬──────┘
       │ 1. 发送请求（带 tools 定义）
       ▼
┌─────────────┐
│   LLM API    │
└──────┬──────┘
       │ 2. 返回 tool_calls（要调哪个函数）
       ▼
┌─────────────┐
│  你的应用代码  │  3. 解析 tool_calls，执行对应函数
└──────┬──────┘
       │ 4. 把函数执行结果发回给 LLM
       ▼
┌─────────────┐
│   LLM API    │  5. LLM 结合函数结果生成最终回答
└──────┬──────┘
       │ 6. 返回最终回答
       ▼
┌─────────────┐
│    用户看到    │
└─────────────┘
```

这里通常会有**两次**模型调用：第一次决定要不要调工具，第二次拿工具结果组织最终回答。

## 用 OpenAI SDK 写一个最小版本

### 1. 定义工具

```typescript
import OpenAI from 'openai';

const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
});

// 定义可用工具
const tools: OpenAI.ChatCompletionTool[] = [
  {
    type: 'function',
    function: {
      name: 'getWeather',
      description: '获取指定城市的当前天气信息',
      parameters: {
        type: 'object',
        properties: {
          city: {
            type: 'string',
            description: '城市名称，如"北京"、"上海"',
          },
        },
        required: ['city'],
      },
    },
  },
  {
    type: 'function',
    function: {
      name: 'searchBlog',
      description: '搜索博客文章',
      parameters: {
        type: 'object',
        properties: {
          query: {
            type: 'string',
            description: '搜索关键词',
          },
          category: {
            type: 'string',
            enum: ['react', 'vue', 'javascript', 'css', 'web'],
            description: '文章分类（可选）',
          },
        },
        required: ['query'],
      },
    },
  },
];
```

工具定义本质上就是 JSON Schema。模型主要靠这些信息判断：
- 工具叫什么名字
- 工具能干什么（description 很重要，模型靠它判断什么时候该用）
- 需要什么参数、参数是什么类型

### 2. 实现工具函数

```typescript
// 实际的工具实现（这些是你的业务代码，模型不会看到）
async function getWeather(city: string) {
  // 实际项目中调用天气 API
  const response = await fetch(
    `https://api.weather.com/v1/current?city=${city}`
  );
  return response.json();
}

async function searchBlog(query: string, category?: string) {
  // 实际项目中查数据库
  const results = await prisma.post.findMany({
    where: {
      content: { contains: query },
      ...(category && { category }),
    },
    take: 5,
  });
  return results;
}

// 工具名到函数的映射
const toolHandlers: Record<string, Function> = {
  getWeather: ({ city }: { city: string }) => getWeather(city),
  searchBlog: ({ query, category }: { query: string; category?: string }) =>
    searchBlog(query, category),
};
```

### 3. 把调用循环串起来

```typescript
async function chat(userMessage: string) {
  const messages: OpenAI.ChatCompletionMessageParam[] = [
    {
      role: 'system',
      content: '你是一个有用的助手，可以查询天气和搜索博客文章。',
    },
    { role: 'user', content: userMessage },
  ];

  // 第一次调用：让模型决定是否需要调用工具
  const response = await client.chat.completions.create({
    model: 'gpt-4o',
    messages,
    tools,
  });

  const choice = response.choices[0];

  // 如果模型决定直接回答（不需要工具）
  if (choice.finish_reason === 'stop') {
    return choice.message.content;
  }

  // 如果模型决定调用工具
  if (choice.finish_reason === 'tool_calls') {
    const toolCalls = choice.message.tool_calls!;

    // 把模型的回复（包含 tool_calls）加入消息历史
    messages.push(choice.message);

    // 执行每个工具调用
    for (const toolCall of toolCalls) {
      const { name, arguments: args } = toolCall.function;
      const parsedArgs = JSON.parse(args);

      console.log(`调用工具: ${name}`, parsedArgs);

      // 执行实际函数
      const result = await toolHandlers[name](parsedArgs);

      // 把工具结果加入消息历史
      messages.push({
        role: 'tool',
        tool_call_id: toolCall.id,
        content: JSON.stringify(result),
      });
    }

    // 第二次调用：让模型根据工具结果生成最终回答
    const finalResponse = await client.chat.completions.create({
      model: 'gpt-4o',
      messages,
      tools,
    });

    return finalResponse.choices[0].message.content;
  }
}
```

### 4. 大概会看到什么效果

```typescript
// 需要工具的问题
await chat('北京今天天气怎么样？');
// → 调用工具: getWeather { city: "北京" }
// → "北京今天天气晴朗，气温 28°C，湿度 45%，适合户外活动。"

// 不需要工具的问题
await chat('什么是 React？');
// → 直接回答，不调用任何工具

// 模型自己判断该用哪个工具
await chat('帮我找找关于 hooks 的文章');
// → 调用工具: searchBlog { query: "hooks", category: "react" }
// → "找到了 3 篇相关文章：..."
```

## 多个工具也可以一起调

有些模型会一次返回多个 `tool_calls`，意思是它觉得这些工具可以同时跑：

```typescript
// 用户: "北京和上海今天天气分别怎么样？"
// 模型返回:
toolCalls = [
  { function: { name: 'getWeather', arguments: '{"city":"北京"}' } },
  { function: { name: 'getWeather', arguments: '{"city":"上海"}' } },
];

// 可以并行执行
const results = await Promise.all(
  toolCalls.map(tc => {
    const args = JSON.parse(tc.function.arguments);
    return toolHandlers[tc.function.name](args);
  })
);
```

类比前端：就像 `Promise.all` 并行请求多个 API，而不是串行等待。

## 工具描述的重要性

模型决定"用不用工具"和"用哪个工具"，**完全依赖你写的 description**。

```typescript
// ❌ 差的描述：模型不知道什么时候该用
{
  name: 'search',
  description: '搜索',
}

// ✅ 好的描述：模型能准确判断使用场景
{
  name: 'searchBlog',
  description: '在博客中搜索技术文章。当用户询问特定技术话题、想找相关文章、或需要代码示例时使用。支持按分类筛选。',
}
```

这其实就是 Prompt Engineering 在工具定义上的体现——description 就是给模型的 prompt。

## 放到我的博客项目里

我现在博客里的 RAG 聊天，最初更像这条固定管线：

```
用户提问 → Embedding 检索 → 拼接 context → LLM 生成回答
```

加上 Function Calling 之后，就可以把“要不要查、查哪里”交给模型先判断：

```
用户提问 → LLM 判断意图
  ├─ "帮我找 React 相关文章" → 调用 searchBlog 工具 → 返回文章列表
  ├─ "React hooks 怎么用" → 调用 RAG 检索工具 → 结合上下文回答
  ├─ "总结一下你有哪些分类" → 调用 getCategories 工具 → 返回分类列表
  └─ "你好" → 直接回答，不调用工具
```

这样就不用所有问题都先走 RAG。比如用户只是打招呼，就没必要把向量库翻一遍。

## Function Calling 和 Agent 差在哪

Function Calling 是**单次决策**：模型看一次就决定调什么工具。

Agent 是**多次决策循环**：模型调完一个工具，看到结果后可能决定再调另一个工具，直到任务完成。

```
Function Calling（单步）：
  用户提问 → 模型决策 → 调用工具 → 生成回答 → 结束

Agent（多步循环）：
  用户提问 → 模型决策 → 调用工具 A → 看结果 → 决定再调工具 B
  → 看结果 → 觉得信息够了 → 生成最终回答 → 结束
```

我会把 Function Calling 当成 Agent 的底层动作能力。它像 `fetch`：能直接用，也能被 React Query 这种更上层的东西封装。LangGraph 对 Agent 来说就有点像这种上层编排。

## 这里我会特别小心的坑

### 1. 模型幻觉参数
模型可能生成不存在的参数值。比如你定义了 `category` 的 enum 是 `['react', 'vue']`，它也可能给你传一个 `'angular'`。所以工具执行前一定要做参数校验。

```typescript
// 用 zod 做运行时校验
import { z } from 'zod';

const searchSchema = z.object({
  query: z.string(),
  category: z.enum(['react', 'vue', 'javascript', 'css', 'web']).optional(),
});

// 在执行工具前校验
const parsed = searchSchema.safeParse(parsedArgs);
if (!parsed.success) {
  // 返回错误信息给模型，让它重新生成
  return { error: parsed.error.message };
}
```

### 2. JSON 解析失败
模型生成的 arguments 偶尔不是合法 JSON。加 try-catch：

```typescript
let parsedArgs;
try {
  parsedArgs = JSON.parse(toolCall.function.arguments);
} catch {
  // 告诉模型参数格式有误
  messages.push({
    role: 'tool',
    tool_call_id: toolCall.id,
    content: JSON.stringify({ error: '参数解析失败，请重新生成' }),
  });
}
```

### 3. 工具太多导致选择困难
给模型太多工具（>20 个），它的选择准确率会下降。解决方案：
- 按场景分组，只传相关的工具
- 用 RAG 检索相关工具（是的，工具本身也可以做检索）
- 分层：先让一个模型选工具类别，再让另一个模型选具体工具
