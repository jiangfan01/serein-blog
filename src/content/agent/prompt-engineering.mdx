# Prompt Engineering

## 本质：你在写"需求文档"

Prompt Engineering 不是什么玄学，本质就是**给 LLM 写需求文档**。你之前和后端对接写接口文档、写 PRD，其实就是同一件事——把你的意图精确传达给一个"不会主动问你"的执行者。

区别在于：后端看不懂你的文档会来问你，LLM 看不懂你的 prompt 会自己瞎编。所以 prompt 要比接口文档写得更精确。

## System Prompt：定义"这个 AI 是谁"

System Prompt 是整个对话的基调，相当于给 LLM 一个角色设定。

```typescript
const messages = [
  {
    role: 'system',
    content: `你是 Serein Blog 的 AI 助手。
你的职责是回答关于前端开发的技术问题。
你只基于博客中的笔记内容回答，不要编造信息。
如果笔记中没有相关内容，直接说"我的知识库中没有这方面的内容"。
回答时使用中文，语气专业但不生硬。`,
  },
  { role: 'user', content: userQuestion },
];
```

### System Prompt 的关键要素

| 要素 | 作用 | 示例 |
|---|---|---|
| 角色定义 | 模型以什么身份回答 | "你是一个前端技术助手" |
| 能力边界 | 什么能做什么不能做 | "只基于提供的上下文回答" |
| 输出格式 | 回答的结构要求 | "用 markdown 格式，包含代码示例" |
| 兜底策略 | 不知道时怎么办 | "如果不确定，说明而不是编造" |
| 语气风格 | 沟通方式 | "专业但不生硬，像同事聊天" |

## 工具描述：决定 Agent 的智商

上一篇讲了 Function Calling，模型选工具完全靠 description。这里深入讲怎么写好它。

### 差的描述 vs 好的描述

```typescript
// ❌ 模型不知道什么时候该用
{
  name: 'search',
  description: '搜索文章',
}

// ✅ 模型能准确判断
{
  name: 'searchBlog',
  description: `在博客知识库中搜索技术文章。
使用场景：
- 用户想找某个技术话题的文章
- 用户问"有没有关于 xxx 的笔记"
- 用户需要特定分类下的内容
不要用于：
- 用户只是闲聊打招呼
- 用户问的是通用知识而非博客内容`,
}
```

### 参数描述同样重要

```typescript
parameters: {
  type: 'object',
  properties: {
    query: {
      type: 'string',
      description: '搜索关键词。提取用户问题中的核心技术术语，如"React hooks"、"虚拟DOM"。不要传入完整的用户问句。',
    },
    category: {
      type: 'string',
      enum: ['react', 'vue', 'javascript', 'css', 'web'],
      description: '文章分类。只在用户明确提到分类时传入，否则不传，让搜索覆盖所有分类。',
    },
  },
}
```

description 里写清楚"什么时候传、什么时候不传、传什么格式"，模型的参数准确率会大幅提升。

## 核心技巧

### 1. Few-shot：给例子比讲道理有效

```typescript
const systemPrompt = `你是一个代码审查助手。

审查规则：
- 检查类型安全
- 检查错误处理
- 检查命名规范

示例输入：
\`\`\`typescript
function get(id) {
  return fetch('/api/' + id).then(r => r.json())
}
\`\`\`

示例输出：
1. ⚠️ 类型安全：参数 id 缺少类型注解，建议 id: string
2. ⚠️ 错误处理：缺少 .catch() 或 try-catch，fetch 失败会导致未处理的 Promise rejection
3. ⚠️ 命名规范：函数名 get 太泛，建议改为 getUserById

请按照以上格式审查用户提供的代码。`;
```

给一个输入-输出的例子，比写十行规则描述都管用。模型通过例子学习模式，比理解抽象规则更准确。

### 2. 结构化输出：用 JSON Schema 约束

当你需要模型返回结构化数据时，明确指定格式：

```typescript
const response = await client.chat.completions.create({
  model: 'gpt-4o',
  messages: [
    {
      role: 'system',
      content: `分析用户描述的 bug，返回 JSON 格式：
{
  "severity": "high" | "medium" | "low",
  "category": "逻辑错误" | "类型错误" | "性能问题" | "样式问题",
  "summary": "一句话描述",
  "suggestedFix": "修复建议"
}`,
    },
    { role: 'user', content: bugDescription },
  ],
  response_format: { type: 'json_object' }, // 强制 JSON 输出
});
```

`response_format: { type: 'json_object' }` 是 OpenAI 的硬约束，保证输出一定是合法 JSON。比在 prompt 里写"请返回 JSON"靠谱得多。

### 3. Chain of Thought：让模型"想一步说一步"

```typescript
// ❌ 直接要答案，容易出错
'这段代码有什么问题？'

// ✅ 让模型分步思考
`请按以下步骤分析这段代码：
1. 先理解代码的意图是什么
2. 逐行检查是否有逻辑错误
3. 检查边界情况和错误处理
4. 给出你的结论和修复建议

一步一步来，每步都写出你的思考过程。`
```

这不是玄学——模型是自回归生成的（上一篇讲过），前面生成的 token 会影响后面的 token。让它先"想"再"答"，相当于给它更多的中间 token 来推理，准确率自然更高。

### 4. 角色扮演：利用模型的训练数据

```typescript
// 通用问法
'帮我优化这个 SQL 查询'

// 角色扮演（效果更好）
'你是一个有 10 年经验的 DBA，擅长 PostgreSQL 性能优化。请从索引、查询计划、数据量三个角度分析这个查询的性能瓶颈。'
```

模型训练数据里有大量"专家回答"的模式，给它一个专家角色，它会倾向于输出更专业的内容。

### 5. 负面约束：告诉模型不要做什么

```typescript
const systemPrompt = `你是博客 AI 助手。

规则：
- 只基于提供的上下文回答
- 不要编造代码示例
- 不要推荐上下文中没有提到的库或工具
- 如果上下文不足以回答，说"我需要更多信息"而不是猜测
- 不要在回答开头说"好的"、"当然"等废话，直接给内容`;
```

模型有"讨好用户"的倾向（训练时的 RLHF 导致的），明确告诉它不要做什么，比只说要做什么更有效。

## 实战：优化你博客的 RAG Prompt

你现在的 RAG 可能是这样的 prompt：

```typescript
// 基础版
const prompt = `根据以下上下文回答问题：
上下文：${context}
问题：${question}`;
```

优化后：

```typescript
const prompt = `你是 Serein Blog 的技术问答助手，专注于前端开发领域。

## 你的知识来源
以下是从博客笔记中检索到的相关内容：

${context}

## 回答规则
1. 只基于上面的笔记内容回答，不要引入外部知识
2. 如果笔记内容不足以回答，明确说"博客中暂时没有这方面的详细笔记"
3. 回答中引用具体的笔记内容时，指出来源（如"在 React Hooks 笔记中提到..."）
4. 代码示例直接使用笔记中的代码，不要自己编造
5. 用中文回答，技术术语保留英文

## 用户问题
${question}`;
```

区别在于：
- 明确了知识边界（只用笔记内容）
- 有兜底策略（不知道就说不知道）
- 要求引用来源（增加可信度）
- 约束了代码示例的来源（防止幻觉）

## Prompt 调试心法

| 问题 | 原因 | 解决 |
|---|---|---|
| 模型回答太啰嗦 | 没有约束输出长度 | 加"用 3 句话以内回答"或"不超过 200 字" |
| 模型总是编造信息 | 没有兜底策略 | 加"如果不确定，说不知道" |
| 模型不调用工具 | description 不够明确 | 在 description 里加使用场景示例 |
| 模型调错工具 | 工具之间职责重叠 | 在 description 里加"不要用于..."的负面约束 |
| 输出格式不稳定 | 只用自然语言描述格式 | 用 JSON Schema 或 response_format 硬约束 |
| 复杂问题回答质量差 | 模型跳步推理 | 加 Chain of Thought，要求分步思考 |

## 和 Agent 开发的关系

在 Agent 系统中，Prompt Engineering 体现在三个层面：

1. **System Prompt** → 定义 Agent 的角色、能力、行为边界
2. **Tool Description** → 决定 Agent 什么时候用什么工具（上一篇的核心）
3. **Orchestration Prompt** → 在多步推理中引导 Agent 的决策逻辑（下一篇 LangGraph 会讲）

写好 prompt 是 Agent 开发的基本功，就像写好 CSS 是前端的基本功——不难，但做好了效果差距巨大。
