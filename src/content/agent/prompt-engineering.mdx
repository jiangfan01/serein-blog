# Prompt Engineering

## 我现在怎么理解 Prompt

Prompt Engineering 这个词听起来很大，但我现在会把它当成：**给 LLM 写一份足够清楚的需求说明**。

区别在于，后端同事看不懂会来问你，模型看不懂大概率不会问，它会自己补剧情。所以 prompt 不是玄学，更多是把边界、输入、输出、兜底讲清楚。

## System Prompt：定义"这个 AI 是谁"

System Prompt 是整个对话的基调，相当于给 LLM 一个角色设定。

```typescript
const messages = [
  {
    role: 'system',
    content: `你是 Serein Blog 的 AI 助手。
你的职责是回答关于前端开发的技术问题。
你只基于博客中的笔记内容回答，不要编造信息。
如果笔记中没有相关内容，直接说"我的知识库中没有这方面的内容"。
回答时使用中文，语气专业但不生硬。`,
  },
  { role: 'user', content: userQuestion },
];
```

### 我会在 System Prompt 里写清楚什么

| 要素 | 作用 | 示例 |
|---|---|---|
| 角色定义 | 模型以什么身份回答 | "你是一个前端技术助手" |
| 能力边界 | 什么能做什么不能做 | "只基于提供的上下文回答" |
| 输出格式 | 回答的结构要求 | "用 markdown 格式，包含代码示例" |
| 兜底策略 | 不知道时怎么办 | "如果不确定，说明而不是编造" |
| 语气风格 | 沟通方式 | "专业但不生硬，像同事聊天" |

## 工具描述会直接影响 Agent 的表现

上一篇看 Function Calling 的时候有个点很明显：模型选工具主要靠 description。description 写得含糊，模型就会乱选；写得像真实使用说明，它就稳定很多。

### 模糊描述 vs 能用的描述

```typescript
// ❌ 模型不知道什么时候该用
{
  name: 'search',
  description: '搜索文章',
}

// ✅ 模型能准确判断
{
  name: 'searchBlog',
  description: `在博客知识库中搜索技术文章。
使用场景：
- 用户想找某个技术话题的文章
- 用户问"有没有关于 xxx 的笔记"
- 用户需要特定分类下的内容
不要用于：
- 用户只是闲聊打招呼
- 用户问的是通用知识而非博客内容`,
}
```

### 参数描述同样重要

```typescript
parameters: {
  type: 'object',
  properties: {
    query: {
      type: 'string',
      description: '搜索关键词。提取用户问题中的核心技术术语，如"React hooks"、"虚拟DOM"。不要传入完整的用户问句。',
    },
    category: {
      type: 'string',
      enum: ['react', 'vue', 'javascript', 'css', 'web'],
      description: '文章分类。只在用户明确提到分类时传入，否则不传，让搜索覆盖所有分类。',
    },
  },
}
```

description 里把“什么时候传、什么时候不传、传什么格式”写明白，工具调用会少很多莫名其妙的问题。

## 几个我会反复用的小技巧

### 1. Few-shot：给例子比讲道理更稳

```typescript
const systemPrompt = `你是一个代码审查助手。

审查规则：
- 检查类型安全
- 检查错误处理
- 检查命名规范

示例输入：
\`\`\`typescript
function get(id) {
  return fetch('/api/' + id).then(r => r.json())
}
\`\`\`

示例输出：
1. ⚠️ 类型安全：参数 id 缺少类型注解，建议 id: string
2. ⚠️ 错误处理：缺少 .catch() 或 try-catch，fetch 失败会导致未处理的 Promise rejection
3. ⚠️ 命名规范：函数名 get 太泛，建议改为 getUserById

请按照以上格式审查用户提供的代码。`;
```

很多时候，给一个输入-输出例子，比写十行抽象规则管用。模型很吃“模式”，你给它一个样板，它就更容易照着走。

### 2. 结构化输出：能约束就别只靠嘴说

当你需要模型返回结构化数据时，明确指定格式：

```typescript
const response = await client.chat.completions.create({
  model: 'gpt-4o',
  messages: [
    {
      role: 'system',
      content: `分析用户描述的 bug，返回 JSON 格式：
{
  "severity": "high" | "medium" | "low",
  "category": "逻辑错误" | "类型错误" | "性能问题" | "样式问题",
  "summary": "一句话描述",
  "suggestedFix": "修复建议"
}`,
    },
    { role: 'user', content: bugDescription },
  ],
  response_format: { type: 'json_object' }, // 强制 JSON 输出
});
```

`response_format: { type: 'json_object' }` 是更硬的约束，比在 prompt 里写“请返回 JSON”靠谱。能用接口约束的地方，就不要只靠一句自然语言。

### 3. 分步分析：别一上来就要最终答案

```typescript
// ❌ 直接要答案，容易出错
'这段代码有什么问题？'

// ✅ 让模型分步思考
`请按以下步骤分析这段代码：
1. 先理解代码的意图是什么
2. 逐行检查是否有逻辑错误
3. 检查边界情况和错误处理
4. 给出你的结论和修复建议

一步一步来，每步都写出你的思考过程。`
```

这不是玄学。模型是自回归生成的，前面生成的内容会影响后面。让它先拆问题，再给结论，通常比直接丢一句“这段代码有什么问题”更稳。

### 4. 角色设定：别写得太戏剧化

```typescript
// 通用问法
'帮我优化这个 SQL 查询'

// 角色扮演（效果更好）
'你是一个有 10 年经验的 DBA，擅长 PostgreSQL 性能优化。请从索引、查询计划、数据量三个角度分析这个查询的性能瓶颈。'
```

角色设定有用，但别写成中二设定。告诉它“你从什么角度看问题”就够了，比如 DBA 看 SQL、前端工程师看组件边界。

### 5. 负面约束：明确哪些事不要做

```typescript
const systemPrompt = `你是博客 AI 助手。

规则：
- 只基于提供的上下文回答
- 不要编造代码示例
- 不要推荐上下文中没有提到的库或工具
- 如果上下文不足以回答，说"我需要更多信息"而不是猜测
- 不要在回答开头说"好的"、"当然"等废话，直接给内容`;
```

模型很容易为了让回答看起来完整而补东西，所以“不要编造”“上下文不足就说明”这种约束要写在前面。

## 放到博客 RAG 里怎么写

最开始的 RAG prompt 可能会写得很短：

```typescript
// 基础版
const prompt = `根据以下上下文回答问题：
上下文：${context}
问题：${question}`;
```

我会把它改成更明确的版本：

```typescript
const prompt = `你是 Serein Blog 的技术问答助手，专注于前端开发领域。

## 你的知识来源
以下是从博客笔记中检索到的相关内容：

${context}

## 回答规则
1. 只基于上面的笔记内容回答，不要引入外部知识
2. 如果笔记内容不足以回答，明确说"博客中暂时没有这方面的详细笔记"
3. 回答中引用具体的笔记内容时，指出来源（如"在 React Hooks 笔记中提到..."）
4. 代码示例直接使用笔记中的代码，不要自己编造
5. 用中文回答，技术术语保留英文

## 用户问题
${question}`;
```

区别主要在这里：
- 明确了知识边界（只用笔记内容）
- 有兜底策略（不知道就说不知道）
- 要求引用来源（增加可信度）
- 约束了代码示例的来源（防止幻觉）

## 调 prompt 的时候我会看这些问题

| 问题 | 原因 | 解决 |
|---|---|---|
| 模型回答太啰嗦 | 没有约束输出长度 | 加"用 3 句话以内回答"或"不超过 200 字" |
| 模型总是编造信息 | 没有兜底策略 | 加"如果不确定，说不知道" |
| 模型不调用工具 | description 不够明确 | 在 description 里加使用场景示例 |
| 模型调错工具 | 工具之间职责重叠 | 在 description 里加"不要用于..."的负面约束 |
| 输出格式不稳定 | 只用自然语言描述格式 | 用 JSON Schema 或 response_format 硬约束 |
| 复杂问题回答质量差 | 模型跳步推理 | 加 Chain of Thought，要求分步思考 |

## 和 Agent 开发怎么接上

放到 Agent 系统里，prompt 大概会出现在三层：

1. **System Prompt** → 定义 Agent 的角色、能力、行为边界
2. **Tool Description** → 决定 Agent 什么时候用什么工具（上一篇的核心）
3. **Orchestration Prompt** → 在多步推理中引导 Agent 的决策逻辑（下一篇 LangGraph 会讲）

写 prompt 有点像写 CSS：语法不难，但边界、层级、默认值没想清楚，最后效果就会很飘。
